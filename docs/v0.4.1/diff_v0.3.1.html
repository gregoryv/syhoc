34§2 Requirement Identifier Syntax
35 
36Requirement identifiers MUST(R1) be placed directly after keywords as
37defined by [RFC 2119].  Requirement identifiers SHOULD(R2) be written
38within parenthesis and start with 'R' followed by a number.  This
39combination is less likely to conflict with normal wording and use of
40parenthesis within a document.  Requirement identifiers MUST(R3) be
41unique within one specification.
42 
43Example;
44 
45    "... MUST NOT(R1) .." ok
46    "... SHOULD ..."      missing identifier
47    "... MAY (R1) ..."    error spacing after keyword
48    "... SHOULD\nNOT(R1)" ok when keyword split over two lines
49 
50 
51It is RECOMMENDED(R4) that authors do not use identifier series for
52grouping, e.g. identifiers within the R100 series are used for
53feature X and R200 series for feature Y.  It is also RECOMMENDED(R5)
54that authors not change identifiers for one requirement over time.
55This is however likely to happen more during authoring and reviewing
56phase but should be avoided once a specification is published.

68§2 Format Rules
69 
70Main purpose of formatting rules is to enable writers and readers to
71focus on content.  Secondary purpose is enabling tools to process
72specifications for improved presentation or automated consumption.
73The [source of this document] is written following these rules.
74 
75 
76§2.1 Sections
77 
78Sections SHOULD(#R1) start with the § sign followed by a level and a
79heading.
80 
81    "§9.1 Example Heading..."
82 
83 
84§2.2 Requirement Identifiers
85 
86Without requirement identifiers it's difficult to reference them.
87Requirement identifiers SHOULD(#R7) be written within parenthesis and
88start with '#R' followed by a number.  Requirement identifiers
89MUST(#R10) be unique within one specification.  Once a requirement
90identifier has been assigned and specification published, it MUST
91NOT(#R11) be reused.  Requirement identifiers MUST(#R8) be placed
92directly after one of the keywords as defined by [RFC 2119].
93 
94    "... MUST NOT(#R999) .."

97§2.3 Links

99§2.3.1 Links to Sections
100 
101Links to sections MUST(#R2) be written within parentheses.
102 
103    "... (§9.1) ..."
104 
105Transformed links to sections SHOULD(#R3) replace the § sign with the
106word 'section'.
107 
108    "... (section 9.1) ..."

111§2.3.2 External Links
112 
113Within a body of test external links and their URL make them less
114readable.  By separating the phrase that should be linked and the URL
115readability is improved.
116 
117External link phrase SHOULD(#R4) be written within square brackets.
118 
119    "... [some phrase] ..."
120 
121URL's of external link phrases SHOULD(#R5) be defined on separate
122lines.  Preferably at the end of a document or section.
123 
124    "[some phrase]: https://example.com"
125 
126Lines defining URL's for external link phrases MAY(#R9) be removed
127from the final document.

130§2.3.3 Internal Reference Links
131 
132Internal reference links SHOULD(#R6) be written within square
133brackets.
134 
135    "... see [1] ..."
136 
137If a internal reference link is a positive integer it refer to a list
138of references in the same document.  In this document see (§3).

141§2.3.4 Internal Requirement Links
142 
143Within a specification requirements are often related and have plenty
144of cross references.  Internal requirement links SHOULD(#R13) be
145written within square brackets.
146 
147   "... see [#R99] ..."
148 
149Internal requirement links MUST(#R12) reference existing requirement.
